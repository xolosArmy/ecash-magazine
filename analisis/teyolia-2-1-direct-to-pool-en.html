The Layer 1 Return Path Rule

Based on the THORChain-side clarification, the ownership question for Teyolia 2.1 is no longer a vague hypothesis. It becomes a concrete architectural rule: the THOR-side owner and the XEC-side return path are not the same thing.

In a dual-sided LP flow, the THOR address used as PAIREDADDR can initiate the withdrawal, but the XEC side is still paid back to the paired Layer 1 address — that is, whatever THORChain considers the sender of the initial XEC ADD transaction.

Therefore, +:XEC.XEC:<thor-community-owner> helps with THOR/RUNE-side pairing and withdrawal authority, but it does not remove the need for a durable, recoverable XEC-side paired L1 address.

Final architecture rule:

Teyolia 2.1 Direct-to-Pool must not treat an ephemeral covenant as the secure owner of a manageable LP position unless that covenant/script is explicitly designed as a durable and recoverable XEC return address with a valid future spend path.

The architecture must strictly separate three elements:

  1. The Withdrawal Initiator: The governance-controlled THOR address used as PAIREDADDR. This address can initiate the LP withdrawal on the THOR/RUNE side.
  2. The Physical XEC Return Path: A durable and recoverable eCash Layer 1 address or script. This is the address to which THORChain would return the XEC side during refunds or withdrawals.
  3. The Finalization Rules: The Teyolia covenant must enforce strict finalize rules: late-bound inbound vault, valid memo policy, correct output ordering, dust safety, fee policy, and a recoverable XEC return path.

This changes the safe architecture from a direct ephemeral covenant payout into a staged, recoverable design: 

Recommended:

Teyolia covenant
→ verifiable finalize rules
→ durable XEC return address / script
→ THORChain inbound vault
→ paired THOR owner for withdrawal authority

The unsafe version is: 

Not recommended for manageable LP:

ephemeral covenant
→ THORChain inbound vault

This means Teyolia 2.1 should be classified into three distinct finalize modes:

  1. Teyolia 2.1-A — XEC-only Direct Add: Experimental only. Not recommended for manageable community LP unless the XEC sender is a durable and recoverable return address.
  2. Teyolia 2.1-B — Dual-Sided / Paired LP: Recommended path for community-managed liquidity. The THOR owner can initiate withdrawal, while the XEC side returns to a durable governance-controlled XEC address or script.
  3. Teyolia 2.1-C — DONATE / Irreversible POL: Possible only if the community explicitly accepts non-withdrawable liquidity. This must be presented as a donation-style flow, not as community-owned LP.

The decisive conclusion is that Direct-to-Pool remains viable, but not as a naive one-step covenant-to-vault payout. The production-grade design requires:

  • late-bound inbound vaults fetched immediately before broadcast,
  • short THORChain-compatible ADD memos,
  • strict UTXO output ordering,
  • a durable XEC paired return address,
  • a governance-controlled THOR withdrawal address,
  • and covenant finalize rules that make all of the above verifiable.

In short: Teyolia 2.1 should separate who can initiate withdrawal from where the XEC physically returns. That separation is the key architectural upgrade from a narrative of direct liquidity into a verifiable L1-to-THORChain design.
