Teyolia 2.1 Direct-to-Pool: a trust-minimized path for bringing XEC into native THORChain liquidity

Teyolia 2.1 Direct-to-Pool is technically viable as an architecture, but not as a minor parameter change to Teyolia 2.0. It requires a new finalize path, late binding for THORChain vaults, explicit LP ownership rules, and a durable XEC Layer 1 return path.

MG Technical Team June 2026 Teyolia · THORChain · eCash

The core idea behind Teyolia 2.1 Direct-to-Pool is simple to express but delicate to implement: allow a community campaign on eCash to accumulate XEC and, once the goal is reached, settle directly into the native XEC/RUNE pool on THORChain without passing through an intermediate custodian.

The technical conclusion is clear: the concept is worth advancing, but only as a new architecture. Direct-to-Pool must not mean “replace the beneficiary with a THORChain address.” It must mean a verifiable finalize path, live THORChain preflight, strict UTXO output ordering, local wallet signing, and explicit rules for who controls the resulting LP position.

The external precondition remains fundamental: XEC must first be officially supported by THORNode/Bifrost and exposed through inbound_addresses with production parameters. Until XEC appears as a supported inbound chain with final dust_threshold, gas_rate, halted status and address format rules, Teyolia 2.1 remains an architectural specification rather than an immediately executable mainnet route.

Precondition 0: XEC must exist on THORChain mainnet first

The design has to separate two planes: feasibility and operational readiness.

At the design level, Teyolia 2.1 is viable in principle. eCash can build UTXO transactions, carry a short memo and allow local wallet signing. THORChain already has a known model for UTXO inbound transactions: send the principal to the current Asgard vault, return change to the VIN0 address, include the memo in OP_RETURN, and let THORNodes observe and process the transaction.

At the operational level, however, Teyolia 2.1 cannot be used in production until XEC is merged, deployed, and exposed as a supported chain by THORChain. The Direct-to-Pool path depends on live network data, not on a static address or a hardcoded assumption.

Teyolia 2.1 Direct-to-Pool is ready as an architecture for implementation once the required THORChain infrastructure preconditions are met.

What changes from Teyolia 2.0

Teyolia 2.0 already improves on a classic custodial crowdfunding model: the campaign exists on eCash, users sign from their wallets, and the backend acts as a transaction-construction helper rather than as the natural owner of the funds.

Teyolia 2.1 pushes this further. The difference is not only where the XEC goes. The difference is who controls the final hop.

In Teyolia 2.0, the final settlement path can still depend on operators or backend policy. In Teyolia 2.1, the contract must constrain the shape of the transaction that is allowed to close the campaign. The contract should no longer say only “pay the beneficiary.” It should say: “finalization is valid only if the transaction has a THORChain-compatible form.”

That requires a contract version bump. It is not a configuration change. It is a new semantic layer.

The correct architecture: eCash accumulation, just-in-time THORChain settlement

The recommended architecture has two planes.

The first plane is eCash. Users contribute XEC into a campaign with explicit rules: goal, expiry, refund policy, finalize mode, fee policy and contract version.

The second plane is THORChain. That plane should activate only at settlement time.

The reason is critical: THORChain inbound vault addresses must not be treated as persistent destinations. They change through churn and operational network conditions. A campaign must not lock an inbound vault address at creation time.

The correct settlement pattern is:

1. Goal reached The campaign reaches its XEC target.
2. Live preflight The builder fetches current XEC inbound_addresses.
3. Network checks The builder checks halted state, LP pause status, gas rate and dust threshold.
4. Build finalize v2.1 The unsigned transaction is constructed with THORChain-compatible outputs.
5. Local signing The wallet signs locally, preserving non-custodial execution.
6. Immediate broadcast The transaction is broadcast immediately; stale attempts must be rebuilt.

The campaign must not commit to a THORChain vault at creation time. It must commit to a valid finalization rule.

The transaction shape THORChain needs

For UTXO chains, THORChain requires a specific transaction shape. The principal must be sent to the current Asgard vault as VOUT0. Change must return to the input address, VIN0, because THORChain identifies the user by VIN0 for refunds. The memo must be placed in OP_RETURN, typically as VOUT2. The transaction must not randomize output ordering and must remain within THORChain output limits.

Translated into Teyolia 2.1, a Direct-to-Pool finalize transaction should look like this:

VOUT0
XEC principal sent to the current THORChain inbound vault.
VOUT1
Change returned to VIN0, if required.
VOUT2
OP_RETURN containing the THORChain ADD memo.
VOUT3+
Optional outputs only if they do not break THORChain parsing, output limits or failure-mode policy.

The liquidity memo uses the form:

ADD:POOL:PAIREDADDR:AFFILIATE:FEE

THORChain also allows the short alias +. The PAIREDADDR parameter links the other side of a dual-sided liquidity position. If no paired address is provided, the action becomes single-sided or pending depending on context.

Minimal test
+:XEC.XEC
Community owner
+:XEC.XEC:<thor-community-owner>
Affiliate LP fee
+:XEC.XEC:<thor-community-owner>:<affiliate-thorname>:100

The minimal memo is useful for integration testing. The community-owner memo is the relevant profile for managed liquidity. The affiliate-fee variant should only be used after THORChain confirms the exact ADDLP affiliate semantics for XEC.

Short memo or nothing

One of the most important constraints is not the maximum payload eCash can carry. It is what THORChain will process safely.

BCH/eCash-style chains can support larger data outputs than Bitcoin Core policy in some contexts. But Direct-to-Pool must be designed for the THORChain UTXO path. The safe production rule is conservative:

Every production memo should fit the short THORChain UTXO path.

Teyolia 2.1 does not need a long memo to prove sovereignty. It needs a correct memo, a valid transaction object and a recoverable return path.

The central issue: who owns the LP position?

The most delicate question is not whether XEC can be sent to THORChain. It is who will be able to withdraw or manage the resulting liquidity later.

THORChain liquidity is permissionless and non-custodial, but liquidity ownership has chain-specific implications. For UTXO inbounds, the L1 sender context matters. In dual-sided liquidity, a THOR address can initiate withdrawal, but the external asset side returns to the paired L1 address associated with the original add.

This means that PAIREDADDR is not enough by itself to make the THOR address the full owner of both sides. It anchors the THOR/RUNE side and the withdrawal authority, but the XEC return path remains tied to the L1 sender that THORChain considers paired with the initial ADD.

For Teyolia, that creates a strict design constraint: a covenant-originated ADD is only safe for managed liquidity if the covenant/script address is intentionally designed as a durable, recoverable XEC return address with a valid spend path for refunds or withdrawals.

The Layer 1 Return Path Rule

The THORChain-side clarification closes the central ownership question for the Teyolia 2.1 design. The result is not merely a memo rule. It is a Layer 1 return-path rule.

Teyolia 2.1 Direct-to-Pool cannot treat an ephemeral covenant as a secure LP owner for managed funds. The architecture must strictly separate three elements:

  1. The Withdrawal Initiator: the governance-controlled THOR address, used as PAIREDADDR, with authority to initiate the withdrawal command.
  2. The Physical Destination — XEC Paired L1: a durable and recoverable eCash Layer 1 address or script, such as a community-controlled multi-signature or policy script, to which THORChain can return the XEC side.
  3. The Finalization Rules: the Teyolia covenant must either finalize through a durable recoverable XEC return address, or itself be programmed with an advanced future spend path capable of recovering returned funds.

Therefore, the memo:

+:XEC.XEC:<thor-community-owner>

helps with dual-sided pairing and withdrawal authority, but it does not remove the need for a recoverable XEC-side paired L1 address.

The safe managed-LP route is not covenant → THORChain vault. It is covenant finalize rules → durable XEC return address/script → THORChain inbound vault → paired THOR owner.

Three Teyolia 2.1 settlement profiles

Direct-to-Pool should not be treated as one feature. It should be treated as a family of settlement profiles, each with different ownership and failure-mode implications.

Teyolia 2.1-A XEC-only asymmetric direct add. Useful for testing or non-managed liquidity, but unsafe for managed LP if the XEC sender is an ephemeral covenant.
Teyolia 2.1-B Dual-sided or pending dual-sided XEC + RUNE. Recommended for community-managed LP, provided the XEC return address is durable and recoverable.
Teyolia 2.1-C Donation-style or POL-style route. Possible only if the community accepts non-recoverable liquidity or protocol donation semantics.

The recommended architecture for managed community liquidity is Teyolia 2.1-B. It separates the XEC return address, the THOR withdrawal initiator and the covenant finalization rules.

The 1% fee: two paths, two philosophies

The 1% infrastructure fee must not be treated as an administrative detail. It must be part of the verifiable settlement design.

1% as protocol LP

The first route is to use THORChain’s native affiliate fee in the ADD memo. In this model, the affiliate is added to the pool as an LP, and the fee is expressed inside THORChain’s own semantics.

Advantage: it is elegant, atomic and aligned with THORChain. The fee only exists if THORChain processes the ADD correctly.

Disadvantage: it should not be assumed to function as immediate liquid XEC treasury. It may become a THORChain-side LP position instead.

1% as liquid XEC treasury

The second route is a Layer 1 split:

VOUT0
99% or net XEC principal sent to the inbound vault.
VOUT1
Change returned to VIN0, if required.
VOUT2
OP_RETURN with the ADD memo.
VOUT3
1% sent to the XEC infrastructure treasury.

Advantage: it is transparent on eCash, easy to audit and useful for day-to-day operations.

Critical risk: if THORChain rejects, delays or refunds the VOUT0 deposit, the VOUT3 fee has already been delivered irreversibly.

For that reason, if Teyolia uses an L1 split, the failure mode must be designed into the contract. It must not be an extra output added casually.

Risks and mitigations

The first risk is external: XEC must reach official THORNode/Bifrost support. Without XEC in inbound_addresses, there is no operational target.

The second risk is contractual: the current finalize path must not be reused as if it were equivalent. Direct-to-Pool needs contractVersion = 2.1, unit tests, reconstruction tests and explicit output rules.

The third risk is LP ownership. If the LP is associated with the wrong L1 identity, liquidity can become difficult or impossible to manage safely.

The fourth risk is operational: vault addresses churn, LP actions can be paused, chains can be halted, and stale settlement attempts can become unsafe.

The fifth risk is economic: asymmetric bootstrap into a new or shallow pool can create high slip and invite immediate arbitrage. A dual-sided or pending dual-sided model is more complex but can produce cleaner ownership and price discovery.

Open questions for THORChain

Before moving from specification to implementation, six points should still be validated with the THORChain technical team:

  1. What are the final acceptance criteria for XEC in THORNode/Bifrost mainnet?
  2. Will the BCH-style long-memo reconstruction path be supported for XEC, or should production XEC flows assume short memos only?
  3. What exact L1 identity will THORChain record as the external paired address for an XEC ADD produced from a covenant/script?
  4. For ADDLP, does the affiliate fee always become LP attribution for the affiliate, or are there liquid distribution paths comparable to swaps?
  5. What final dust_threshold, max-output, address-format and output-ordering rules will apply to XEC?
  6. Will an official quote or estimate endpoint for XEC add-liquidity be available from day one?

Implementation sketch: Teyolia Finalize v2.1 Builder

The next technical deliverable should not be a complex production contract immediately. It should be a strict builder/preflight specification.

Inputs
campaignId, covenant UTXO, durable XEC return address/script, THOR paired owner address, pool asset XEC.XEC, fee policy, live inbound_addresses, dust threshold, gas rate and halted/paused status.
Outputs
An unsigned eCash transaction with VOUT0 as the current XEC inbound vault, VOUT1 as change to VIN0 if required, VOUT2 as the ADD memo, and optional fee outputs only if policy allows them.
Security rule
The builder is not an authority. It is a helper. The contract and wallet must reject invalid finalize shapes.

Implementing Sovereignty

Teyolia 2.1 Direct-to-Pool can become a structural improvement over Teyolia 2.0, but only if it is treated as a new settlement architecture.

The goal is not merely to send funds “to a THORChain address.” The goal is to encode the final hop correctly: which outputs are valid, which memo is allowed, when the live vault is fetched, how fees are handled, who can initiate withdrawal, where XEC returns, and what happens if THORChain is not ready to process the deposit.

Direct-to-Pool must not hardcode a vault. It must hardcode a sovereignty rule.

Teyolia 2.1 will not be more trust-minimized because it promises that funds go directly into a pool. It will be more trust-minimized if the contract prevents them from going anywhere else, if the builder only assists, if the wallet signs locally, and if the final state can be verified from chain data.

XEC is the money.
Liquidity must be native.
And the final hop cannot depend on trust: it must be written into the contract.
